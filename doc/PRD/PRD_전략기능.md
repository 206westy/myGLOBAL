## 0. 비판적 재평가 — 4회 검증

본 PRD는 v1.0 → v2.0 → v3.0의 진화 과정에서 발견된 결함을 반영하여, 4번째 검증 단계를 거쳐 확정된 최종안이다.

### 검증 1 — 정체성 (대시보드 vs 전략 플랫폼)

현재 구현은 *KPI 카드 + 칸반보드* 의 조합으로, 본질적으로 *상태를 보여주는 도구* 다. 사용자가 들어와서 *데이터 보고 떠나는* 화면이며, *결정과 액션을 유도* 하지 않는다. 매니저 평균 체류 시간 30초로 추정.

**결론:** 대시보드가 아닌 *처리 흐름의 도구* 가 되어야 한다.

### 검증 2 — 워크플로우 충실성

10 STEP 중 화면에 노출된 단계가 *3개* 뿐. 나머지 7단계는 *문서로만 존재* 하고 *제품으로는 존재하지 않는* 상태. 가장 중대한 결함.

**결론:** 10 STEP 모두 화면에 명시적으로 노출되어야 한다.

### 검증 3 — Journey 분기 가시성

Journey B(CCB 자동 매칭)가 진입점 자체가 없음. 이는 CIP 가치의 핵심 — *한 번의 발굴이 N개 라인으로 확산* — 흐름이 화면에서 사라진 상태.

**결론:** Journey B는 워크플로우 안에 자연스럽게 통합되어야 한다.

### 검증 4 — 자율성 vs 강제성 (v3.0 재검증)

v3.0이 제안한 *Action Queue 메인* 구조는 모던하나, **CS팀 전체가 함께 쓰는 표준 도구** 라는 본 플랫폼의 본질과 충돌한다.

| 항목 | Action Queue 단독 (v3.0) | 탭 강제 + 큐 결합 (v3.1) |
|------|------------------------|------------------------|
| 시니어 매니저 효율 | 높음 | 중간 |
| 신입 매니저 학습 | 어려움 | 쉬움 (탭 순서대로) |
| 팀 표준화 | 낮음 | 높음 |
| 인수인계 비용 | 높음 | 낮음 |
| 처리 패턴 측정 | 어려움 | 쉬움 |
| 글로벌 사이트 확장 | 어려움 | 쉬움 |

**결론:** 본 플랫폼은 *팀 공용 표준 도구* 이므로 워크플로우 강제가 우선되어야 한다. 단 100% 강제는 또 다른 비효율을 만들기에, **탭으로 순서를 강제하되 시급도 예외 룰을 둔다**.

## 1. 본질 재정의

### 1.1 한 줄 정의

**"CS팀이 같은 흐름으로 일하게 강제하면서, 각 단계 안에서는 결정을 끌어내는 전략 워크스페이스"**

### 1.2 핵심 원칙 5가지

#### 원칙 ① — 탭이 워크플로우를 강제한다 (Forced Sequence)

상단 6개 탭이 워크플로우 순서를 *시각적으로 강제* 한다. 매니저는 왼쪽부터 오른쪽으로 자연스럽게 이동하며 일하게 된다. 모든 매니저가 같은 동선으로 작업하므로 팀 표준화가 보장된다.

#### 원칙 ② — 탭 안은 액션 큐다 (Queue Inside)

각 탭 안에서는 처리해야 할 의사결정 카드가 우선순위순으로 줄세워진다. 데이터 보여주기가 아닌 *결정 처리* 가 본질이다.

#### 원칙 ③ — 모든 카드는 3지선다 결정을 요구한다 (One Card, One Decision)

카드는 *진행 / 보류 / 거절* 또는 그에 준하는 정확히 *3개 옵션* 중 하나를 선택하게 한다. *검토하세요* 가 아니라 *고르세요* 다.

#### 원칙 ④ — 시급도가 순서를 깰 수 있다 (Urgency Override)

탭 위에 빨간 점이 붙으면 (HIGH 우선순위 / SLA 임박) 그 탭을 *우선 처리* 한다. 강제와 자율성 사이의 균형을 잡는 안전장치.

#### 원칙 ⑤ — 통계·AI는 카드 안에서 결정 근거로 작동한다 (Embedded Intelligence)

별도 분석 탭으로 분리하지 않는다. 통계·AI 결과는 카드 펼치기 영역에서 *왜 이걸 처리해야 하는가* 를 설명하는 도구로 사용된다.

### 1.3 비유

본 플랫폼은 *공장의 컨베이어 벨트 + Linear의 Inbox* 정신을 결합한다. 컨베이어가 *흐름을 강제* 하고, 각 작업대에서 *처리할 항목이 큐로 쌓이는* 구조. 들어와서 → 왼쪽 탭부터 → 큐 위에서부터 → 처리하면 → 오른쪽으로 흘러간다.

## 2. 사용자 정의

### 2.1 역할별 워크 패턴

| 역할 | 평일 일과 | 진입 패턴 |
|------|----------|----------|
| **CS 매니저** | 출근 → Detect 탭 → Investigate → ... → Deploy 까지 차례로 | 하루 3~5회, 회당 20~40분 |
| **CS 엔지니어** | 본인 담당 카드만 본인 작업 탭에서 처리 | 현장 / 사무실 혼재, 1일 평균 5~10건 |
| **부서장 / 임원** | 사이드바의 인텔리전스 영역 직행 | 주 1회, 분기 보고 시점 집중 |

### 2.2 권한 모델

- **엔지니어**: 본인 담당 라인 카드만 노출, 작업형 탭 (Investigate / Validate / Deploy 일부) 액세스
- **매니저**: PEE T-CS 전체 카드 + 모든 탭 권한
- **임원 / 부서장**: 매니저 뷰 + 인텔리전스 영역 + 보고서 자동 생성

RLS 기반으로 백엔드 강제.

### 2.3 비목표 (Non-Goals)

본 PRD는 다음을 *명시적으로 다루지 않는다*. 개발자가 자율 판단한다.

- 반응형 레이아웃 (데스크톱 웹 전용 가정)
- 모바일 / 태블릿 최적화
- 다크 모드 / 라이트 모드 전환 정책
- 다국어 i18n 시스템 (한국어 기본, 영문 탭명)

## 4. 6탭 상세 명세

각 탭은 *그 단계에서 처리해야 할 의사결정 카드의 큐* 다. 탭 안은 시급도에 따라 3섹션으로 그룹화된다.

### 4.1 Detect — 발굴 탭

**묶이는 STEP:** 01 (이상 감지) + 02 (CIP 등록)

**탭의 본질적 질문:** *"무엇을 CIP로 올릴 것인가?"*

#### 큐 구성

```
┌─ 🚨 긴급 처리 (HIGH 우선순위 신규 Alert) ─────────────┐
│  HIGH  ZEDIUS XP / SOURCE 호출 +96%                  │
│        SUPRA-N · SEC-X2L(CN)                         │
│        💡 권고: CIP 생성 (위험도 87)                  │
│        [CIP 생성] [계속 Watch] [반려]                  │
└──────────────────────────────────────────────────────┘

┌─ 🔬 일반 처리 (MED Alert + Watch 승격 후보) ────────┐
│  ... 카드들                                          │
└──────────────────────────────────────────────────────┘

┌─ 📋 참고 (현재 Watch 중인 항목) ────────────────────┐
│  ... 카드들 (액션 없음, 추적만)                        │
└──────────────────────────────────────────────────────┘
```

#### 매니저 액션 (3지선다)

| 옵션 | 결과 | 다음 큐 |
|------|------|--------|
| ① CIP 생성 | CIP 번호 발급 + 담당자 배정 | Investigate 탭 |
| ② 계속 Watch | 사유 입력 → 다음 달 재평가 | 동일 탭 (참고 섹션) |
| ③ 반려 | 사유 입력 → 종결 | 사라짐 |

#### 결정 근거 (펼치기)

- **차트:** 호출빈도 추이 / CUSUM 누적 / Pareto 분포
- **수치:** 6개월 이동평균, 표준편차, 영향 설비 비율
- **정성 힌트:** LLM이 S/O 디스크립션에서 추출한 패턴 (반복고장 / 연쇄고장 / SOP 부재)
- **관련 S/O:** 해당 그룹의 최근 서비스 오더 목록

#### 탭 내 보조 도구

상단 우측에 *"+ 수동 생성"* 버튼 (자동 감지를 거치지 않은 매니저 직접 등록).
*"스크리닝 결과 전체 보기"* 링크 (조회 모드, 액션 없음).

**묶이는 STEP:** 03 (현장 조사)

**탭의 본질적 질문:** *"근본 원인이 무엇이고, 솔루션 단계로 갈 만한가?"*

```

│  [Solve 단계 진입] [추가 조사 요청] [반려]              │
└──────────────────────────────────────────────────────┘

┌─ 🔬 일반 처리 (조사 완료 검토 대기) ──────────────────┐
│  ... 카드들                                           │
└──────────────────────────────────────────────────────┘

┌─ ⏳ 진행 중 (엔지니어 작업 중, 매니저 액션 없음) ───┐
│  ... 카드들 (참고용)                                   │
└──────────────────────────────────────────────────────┘
```

#### 매니저 액션 (3지선다)

| 옵션 | 결과 | 다음 큐 |
|------|------|--------|
| ① Solve 단계 진입 | 솔루션 탐색 시작 | Solve 탭 |
| ② 추가 조사 요청 | 엔지니어 재배정 | 동일 탭 (진행 중) |
| ③ 반려 | 사유 입력 → 종결 | 사라짐 |

#### 엔지니어 액션 (작업 모드)

엔지니어 권한으로 진입 시 *작업 카드* 가 노출된다.

- 5 Why 체인 입력
- Fishbone (6M) 구조화
- 8D D0~D4 템플릿
- 사진 / 로그 / 모니터링 데이터 첨부
- 결론: 확정 / 기각 / 추가조사 필요

#### AI 보조 — 조사 품질 검토

LLM이 5 Why / Fishbone 입력을 받아 *논리 약점 검토* 자동 코멘트.
예시: *"Why 3에서 Why 4로 가는 인과 연결이 약합니다. '왜 보드 수명 기준이 없는가?' 가 더 깊은 원인일 수 있습니다."*

### 4.3 Solve — 솔루션 탭

**묶이는 STEP:** 04 (CCB 매칭) + 05 (AI 발굴)

**탭의 본질적 질문:** *"기존 솔루션을 쓸 것인가, 새로 개발할 것인가?"*

#### 큐 구성 — 두 진입 경로

```
┌─ 🎯 CCB 매칭 결과 검토 (STEP 04) ─────────────────────┐
│  CIP-2026-0012  EFEM Joint Board                     │
│  매칭 솔루션 3개 발견 (최대 유사도 87%)                │
│  💡 권고: 기존 솔루션 적용 → Deploy 직진              │
│  [기존 솔루션 적용] [신규 발굴 진행] [부분 차용]        │
└──────────────────────────────────────────────────────┘

┌─ 🤖 AI 솔루션 결과 승인 (STEP 05) ───────────────────┐
│  CIP-2026-0008  RF Cable 품질개선                     │
│  AI가 3가지 솔루션 후보 + TRIZ 모순 매트릭스 분석      │
│  [연구소 이관 승인] [재추론 요청] [폐기]               │
└──────────────────────────────────────────────────────┘
```

#### 매니저 액션 (STEP 04 — CCB 매칭)

| 옵션 | 결과 | 다음 큐 |
|------|------|--------|
| ① 기존 솔루션 적용 | Journey B 합류, 연구소 단계 건너뜀 | **Deploy 탭** (Develop / Validate 스킵!) |
| ② 신규 발굴 진행 | AI 솔루션 발굴 시작 | 동일 탭 (STEP 05) |
| ③ 부분 차용 | 기존 솔루션 일부 + 신규 보완 | 동일 탭 (STEP 05, 컨텍스트 포함) |

#### 매니저 액션 (STEP 05 — AI 솔루션)

| 옵션 | 결과 | 다음 큐 |
|------|------|--------|
| ① 연구소 이관 승인 | 기술 제안서 자동 전송 | Develop 탭 |
| ② 재추론 요청 | AI에 추가 컨텍스트 제공 | 동일 탭 |
| ③ 폐기 | 사유 입력 → 종결 | 사라짐 |

#### 엔지니어 작업 — AI 대화 인터페이스

- AI와 *증상 → 원인 → 솔루션* 단계별 추론 대화

- **AI 솔루션:** TRIZ 모순 매트릭스 결과 / Multi-hop RAG 추론 트레이스 / 온톨로지 인과 그래프

### 4.6 Deploy — 확산 탭

**묶이는 STEP:** 10 (확산 진행률) + Journey B 전체

**탭의 본질적 질문:** *"이 솔루션을 어디 어디에 퍼뜨릴 것인가?"*

#### 큐 구성 — 두 진입 경로 통합

```
┌─ 🎯 신규 자동 매칭 결정 대기 (Journey B 진입점) ─────┐
│  CCB-2026-Q1-018  RF Cable 품질개선 (검증완료)        │
│  적용 가능 라인 12곳 자동 매칭됨                       │
│  매칭 신뢰도: 평균 87% / 최저 72%                      │
│  💡 권고: 8곳 일괄 배포 + 4곳 검토                     │
│  [매칭 검토] [일괄 배포 승인] [선택 배포]               │
└──────────────────────────────────────────────────────┘

┌─ 📊 확산 진행 중 (STEP 10) ───────────────────────────┐
│  CIP-2026-0001  RF Cable 적용 8/12 (67%)             │
│  미적용 4곳 — 담당자 리마인더 필요                     │
│  [리마인더 발송] [확산 종료] [상태 보고서]              │
└──────────────────────────────────────────────────────┘

┌─ 🔄 추가 매칭 — 신규 설비 등록 ───────────────────────┐
│  신규 등록 설비: SUPRA-N / SK-NEW1                    │
│  적용 가능 CIP 3건 발견                                │
│  [매칭 검토] [전체 배포]                                │
└──────────────────────────────────────────────────────┘
```

#### Journey B 통합 — 본 탭의 가장 큰 변화

기존 v2.0에서 *별도 탭* 으로 노출하려 했던 Journey B를, **Deploy 탭의 첫 섹션** 으로 통합한다. 이유는:

- Journey B의 본질은 *기존 검증된 솔루션을 다른 라인에 배포* — 즉 **확산** 그 자체
- Validate를 거친 신규 CIP의 STEP 10도, CCB 자동 수집된 외부 솔루션의 Journey B도, 결국 *어디 어디에 적용할까* 라는 같은 결정
- 별도 탭으로 분리하면 *발굴은 자기 일, 확산은 다른 일* 로 인지되는 부작용. 통합하면 *확산이 일과의 한 부분* 으로 자연스러움

#### 매니저 액션 (Journey B 매칭)

| 옵션 | 결과 | 다음 액션 |
|------|------|---------|
| ① 일괄 배포 승인 | 적용 가능 라인 전체에 CIP 카드 일괄 생성 | 각 설비별 Validate 큐 진입 |
| ② 선택 배포 | 체크박스로 특정 라인만 선택 | 선택된 설비만 Validate 큐 |
| ③ 보류 | 사유 입력 → 다음 분기 재검토 | 동일 탭 (보류 섹션) |

#### 결정 근거 (펼치기)

- **매칭 결과 트리:** 국가 → Fab → 라인 → 설비 계층 구조
- **매칭 신뢰도:** 설비별 0~100점
- **4-Tier 우선순위:** 안전 > 수율 > 신뢰성 > 비용
- **영향 분석:** 적용 시 예상 절감 + 적용 리스크

### 5.1 카드 6요소

```
┌──────────────────────────────────────────────────┐

│ ──────────────────────────────────────────────── │
│ 💡 권고: CIP 생성 (영향 설비 43%, CUSUM UCL 돌파) │  ← AI Decision Suggestion
│ ──────────────────────────────────────────────── │
```

### 5.2 카드 시각 강조 룰

- **HIGH 우선순위**: 좌측 3px 빨강 보더
- **SLA D-3 이내**: 우측 상단 D-N 텍스트 빨강
- **SLA 초과**: 카드 전체 옅은 빨강 배경
- **신규 (24h 이내)**: 우측 상단 NEW 점

### 5.3 AI Decision Suggestion (권고 1줄)

각 카드 액션 버튼 위에 AI 권고 1줄이 항상 노출된다.

> 💡 *"권고: CIP 생성 — 영향 설비 43% (18/42), CUSUM UCL 돌파, 정성 힌트 5건"*

**중요한 설계 원칙 — 권고는 결정을 대신하지 않는다.**

- 권고와 다른 결정 시 *사유 입력 모달* 자동 노출
- 입력된 사유는 LLM 학습 데이터로 축적
- 시간이 지날수록 권고 정확도가 향상되는 *집단 지능 루프* 형성

## 6. 결정 근거 영역 — 카드 펼치기 4탭

카드 본체 클릭 시 *별도 모달이나 페이지 이동 없이* 인라인으로 펼쳐지는 영역. 결정에 필요한 모든 근거를 *현재 위치에서* 본다.

### 6.1 4탭 구조

**탭 1 · Evidence (근거)**

해당 STEP의 핵심 근거 데이터.
- Detect: 호출빈도 추이 / CUSUM / Pareto / 정성 힌트
- Solve: CCB 매칭 결과 / TRIZ 분석 / AI 추론 트레이스
- Validate: 통계 검정 결과 / SEMI KPI 비교 / 분포 변화
- Deploy: 매칭 트리 / 신뢰도 점수 / 영향 분석

**탭 2 · Context (컨텍스트)**

이 카드와 관련된 주변 정보.
- 관련 S/O 목록 (최근 6개월)
- 동일 라인 과거 이력
- 동일 모델 다른 라인 비교
- 관련 CCB / 기술 문서 링크

**탭 3 · Impact (영향)**

이 결정이 미치는 파급 효과.
- 영향 설비 비율
- 추정 손실 비용 (현재)
- 확산 시 매출 / 절감 시뮬레이션
- 헬스 스코어 변화 예측

**탭 4 · Activity (활동)**

이 카드의 시간축 이력.
- 코멘트 / 멘션 (Slack 스타일)
- 상태 변경 이력
- 첨부 파일 / 사진 / 로그

### 6.2 펼치기 인터랙션

- 펼치기는 *애니메이션 0.2초 이내* (모던 UX 표준)
- 동시에 여러 카드 펼치기 가능 (비교 검토 용도)
- 펼친 상태에서 액션 버튼 누르면 처리 후 자동 접힘

## 7. 사이드바 — 보조 영역

탭이 *처리하는 곳* 이라면, 사이드바는 *분석하는 곳* 과 *조망하는 곳* 이다.

### 7.1 인텔리전스 (분석하는 곳)

#### 7.1.1 CIP ROI

분기별 / 연도별 CIP가 만들어낸 가치.
- CIP 매출 기여 (CIP 단가 차이 × 적용 라인 수)
- 고객 절감 추정 (브레이크다운 비율 변화 × 웨이퍼 가격)
- 임원 보고용 한 페이지 자동 생성

#### 7.1.2 설비 헬스 스코어

설비 단위 종합 건강도.
- 호출빈도 / 리워크율 / MTBF / 작업시간 가중 합산 (0~100)
- 시계열 추이 + 이상 변화 표시
- 국가 / Fab / 모델별 슬라이스

#### 7.1.3 효과성 분석

CIP별 도입 전후 통계 검정 결과 모음.
- Paired t-test / Wilcoxon / Synthetic Control / DiD
- SEMI E10 KPI 변화
- 매니저 보고 자료 자동 생성 / 다운로드

### 7.2 워크플로우 보조 뷰 (조망하는 곳)

#### 7.2.1 칸반 (전체 7컬럼)

전체 CIP 흐름을 *조감* 할 때. 6탭이 *나의 일* 이라면 칸반은 *우리 팀의 일*.
- 7컬럼: 접수 / 조사 / 솔루션 / 연구소 / 테스트 / 확산 / 완료
- 카드 정보 밀도는 메인 탭과 동일

#### 7.2.2 타임라인 (간트)

연구소 단계 마일스톤 추적 전용. SLA 위반 위험 시각화.

### 7.3 설정 (관리하는 곳)

- CSV 임포트 (mySERVICE SAP 데이터)
- 스크리닝 실행 (자동 / 수동 트리거)
- 룩업 테이블 관리 (lu_part_groups 등)
- 사용자 / 권한 관리 (RLS)

## 8. AI / 통계 / 기술 강화 — 워크플로우 내재형

10 STEP 워크플로우를 벗어나지 않으면서, 각 STEP의 결정 품질을 높이는 기술 요소.

### 8.1 Detect 강화

| 기술 | 도입 효과 |
|-----|---------|
| BOCPD (Bayesian Online Changepoint Detection) | CUSUM이 놓치는 비선형 변화점 감지 |
| LSTM Autoencoder | 다지표 시계열 정상 패턴 학습 후 재구성 오차로 이상 탐지 |
| Causal Impact (Bayesian Structural Time Series) | PM 일정 / 레시피 변경 등 외생 변수 통제 |
| LLM 정성 힌트 추출 | S/O 디스크립션에서 *반복 / 연쇄 / SOP 부재* 자동 추출 |
| 우선순위 학습 모델 | 매니저 과거 결정 패턴 학습 → 위험도 가중치 개인화 |

### 8.2 Solve 강화

| 기술 | 도입 효과 |
|-----|---------|
| BM25 + SBERT + RRF + Cross-encoder | CCB 매칭 정확도 / 재현율 양립 |
| BGE-M3 다국어 임베딩 | 한영 혼용 디스크립션 대응 |
| HyDE (Hypothetical Document Embeddings) | 짧은 쿼리 매칭 정확도 향상 |
| TRIZ 모순 매트릭스 (40 inventive principles) | 공학 모순 해결 패턴 자동 매핑 |
| Multi-hop RAG (HopRAG, RT-RAG) | 다단계 추론 트리 분해 |
| 설비 온톨로지 (TypeDB) | EFEM / TM / PM 모듈 인과 추론 |
| Self-Refine | LLM 1차 답안 자가 비평 → 개선 |

### 8.3 Develop 강화

| 기술 | 도입 효과 |
|-----|---------|

### 8.4 Validate 강화

| 기술 | 도입 효과 |
|-----|---------|
| Paired t-test / Wilcoxon | 챔버 단위 전후 비교 |
| Synthetic Control Method | 미적용 챔버를 가중평균 대조군으로 사용 |
| Difference-in-Differences | 다른 라인 변화 통제하고 CIP 효과만 분리 |
| Bayesian Causal Impact | 신뢰구간 포함 효과 추정 |

### 8.5 Deploy 강화

| 기술 | 도입 효과 |
|-----|---------|

## 12. 리스크 및 대응

### 12.1 강제 워크플로우의 시니어 매니저 저항

시니어가 *내 방식대로* 일하던 패턴을 *왼쪽부터 차례로* 로 강제하면 초기 반발 가능.

**대응:**
- 빨간 점 시급도 예외 룰로 *완전 강제* 가 아닌 *순서 권장* 임을 명시
- 매니저 1:1 온보딩 세션 (30분)
- 첫 1주는 *기존 칸반과 병행* 가능하게 토글 제공, 2주차부터 6탭 디폴트

### 12.2 빈 탭 동기 이슈

특정 탭이 자주 비어있으면 *왜 이 탭이 있나?* 의문 발생.

**대응:**
- 비어있는 탭은 회색 처리 + *"처리할 항목 없음"* 명시
- 비어있어도 *"이 탭에서 다루는 일"* 짧은 설명 노출

### 12.3 카드 결정 부담

3지선다라도 매번 결정하는 게 피로할 수 있음.

**대응:**
- AI Decision Suggestion으로 매번 *권고 + 근거 1줄* 제공 → 결정 부담 경감
- 권고 따르면 1클릭, 다른 결정 시에만 사유 입력 → 빠른 처리 가능
- 일괄 처리 단축키 (선택 카드들에 동일 액션)

### 12.4 Journey B 데이터 부족 (초기)

CCB 자동 수집 시스템이 안정화되기 전엔 Deploy 탭의 첫 섹션이 비어있을 수 있음.

**대응:**
- 수동 CCB 등록 기능 병행 제공
- 시뮬레이션 매칭 모드 (테스트용)
